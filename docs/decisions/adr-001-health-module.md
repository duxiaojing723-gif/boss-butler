# ADR-001: 健康助理模块架构

**状态**：已接受
**日期**：2026-03-12
**关联 task**：health-module

---

## 背景

用户希望在现有管家应用中增加健康记录功能（运动+饮食），需要决定照片上传方式、热量计算方式、以及数据存储方案。

---

## 决策

1. **照片上传**：复用现有 base64-in-JSON 模式，客户端 Canvas 压缩后直接 POST
2. **食物识别**：GPT-4o Vision `detail: 'low'`，通过现有 `_llm.js` 调用
3. **运动热量**：硬编码 MET 公式，不调用 LLM
4. **数据存储**：Supabase 新增 exercises + meals 两张表
5. **导航**：底部 tab 新增第 4 项"健康"

---

## 理由

- base64 模式已在语音功能中验证，无需新依赖
- GPT-4o Vision low detail 每张仅 ~$0.0005，成本可控
- MET 公式足够准确且零延迟零成本
- 直接用 Supabase 保持技术栈一致

---

## 替代方案（为什么没选）

| 方案 | 为什么没选 |
|------|---------|
| Supabase Storage 存图 | MVP 不需要，base64 存 data URL 更简单 |
| 第三方营养 API | 增加依赖和成本，GPT-4o 已够用 |
| Apple Health 集成 | Web 应用无法访问 HealthKit，需原生壳 |

---

## 后果

**正面**：
- 零新依赖，复用现有架构
- 实现快，4 个 task 可完成

**负面 / 风险**：
- base64 图片存 DB 会增大数据量（MVP 可接受）
- AI 营养估算不精确（UI 标注"仅供参考"）
